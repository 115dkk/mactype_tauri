# Control Center CI and release

The Control Center uses independent required checks so a failure identifies the broken contract.

1. `Build and package` builds the pinned public x86/x64 core and fixed service-injector, runs the standalone Rust 신식 서비스 contract and hosted SCM/marker lifecycle, then compiles the Preview Helper, frontend, Tauri executable, service payload, and Inno Setup installer. The hosted marker gate requires both architectures to load their DLL from the active protected runtime directory and cross-checks each marker PID, session, profile digest, and common runtime generation against persisted service telemetry.
2. `Frontend window gallery` opens every public view, including the settings-file workflow, in all ten supported locales at 390, 768, and 1280 pixels. Browser interactions also import a discovered legacy profile with consent, use the native-import adapter, save and apply a profile, exercise allowed legacy-service transitions, switch profile categories, reveal advanced LCD settings, add a font-specific entry, open include/exclude editing, change the preview background, and prove that a language choice persists. JavaScript exceptions, console errors, renderer crashes, missing readiness markers, wrong text direction, and horizontal overflow fail the job. Localized screenshots and Playwright traces are retained for human review.
3. `Lint gates` runs frontend, Rust, Preview Helper C++, and `service-injector` as blocking jobs, rejects generated settings sources that differ from `shared/settings-schema.json`, and verifies exact message-key and placeholder parity plus every generated setting across all ten locale catalogs. The fixed injector builds and tests x86/x64 Release configurations and repeats the production, inherited-handle launcher, and contract targets under MSVC `/analyze`. The legacy core boundary is documented in `docs/lint-policy.md`.

CodeQL is intentionally separate from the required push and pull-request checks. It runs every Monday at 04:00 UTC or by manual dispatch. The workflow analyzes the existing C++ core through an explicit x64 MSVC build, compiles both x86 and x64 `service-injector` production targets inside the same C/C++ analysis database, and analyzes the TypeScript frontend without a build step. Rust remains covered by blocking `rustfmt`, Clippy, and tests.

`.github/workflows/open-service-disposable-vm.yml` is a separate `workflow_dispatch`-only machine-integration harness. It requires a confirmed, labeled self-hosted disposable Windows VM and has no push, pull-request, schedule, or reusable-workflow trigger. Lifecycle, reboot Auto start, AppInit conflict, migration receipt, and multi-session evidence are uploaded as JSON. Until a scenario has a retained dispatch artifact its result is `UNKNOWN`, even when the verifier is implemented.

The profiles window smoke is stronger than a launch check: it points the app at an x86 fixture DLL with the public `IControlCenter` ABI, waits for Helper IPC plus WIC PNG output, force-terminates the child, and requires a second PNG after automatic restart. It then duplicates the fixture profile, changes a scalar, an `[Individual]` row, and an excluded module, atomically saves and reopens the copy, and only then writes the ready marker. Frontend smoke failures write their error into the marker so CI reports the actual failed contract instead of a generic timeout.

The execution window smoke creates the real Tauri tray icon, queries HKCU startup, SCM service state, and both AppInit registry views, then reports ready. Gallery tests exercise the non-admin startup, manual-launch controls, and a fake legacy-service state machine without changing a runner service. Rust tests require foreign, inaccessible, registry-conflicting, and deletion-pending states to expose no unsafe mutation capability.

The Windows build also releases a suspended stampede of identical Control Center processes. It fails unless exactly one process survives, all secondaries exit successfully, every secondary reaches the first instance callback, and the callback restores the existing window. Static policy checks keep the main executable `asInvoker` and `uiAccess=false`, while the machine installer is fixed at `PrivilegesRequired=admin` and the fixed Program Files destination. The normal Control Center remains non-elevated; only installer and explicitly requested machine-service broker operations cross the administrative boundary.

Velopack is not used. The administrator-elevated installer contains the open Control Center, x86 Preview Helper, source-built x86/x64 Detours core and MacLoader, newly authored profile/translations, and a fixed 신식 서비스 payload. After files are staged, it synchronously runs the protected `bootstrap-install` broker; a successful fresh install publishes the exact bundled default profile and returns only after the fixed service is Auto/LocalSystem/Running with strict Ready health. Existing protected ProgramData profile generations survive upgrades and uninstall. The installer never copies Delphi GUI programs, MacTray, existing upstream profiles/translations, updater binaries, or registry-mode settings. It also leaves LocalAppData theme, locale, recent-profile, applied-profile, and profile files byte-for-byte untouched.

Hosted installer CI performs a real baseline install, then runs a test-only failing upgrade whose signed manifest deliberately launches a non-service executable as the new service generation. The installer must return nonzero while the baseline service image, active runtime pointer, protected profile, Ready health, app executable, and protected broker remain usable. That fixture is never uploaded. CI then installs a distinct successful current generation, verifies profile preservation, checks visible failure when the uninstall broker is unavailable, proves exact-owned removal, and exercises foreign fixed-name and 레거시 서비스 blocked-skip paths without mutation.

The installer file and the installed frontend executable both use the stable filename `MacType Control Center.exe`. The application version remains in Windows metadata and release metadata, not in the user-facing `.exe` filename. GitHub Releases normalize spaces in direct asset names, so releases publish `MacType-Control-Center.zip`; the installer inside retains the exact stable filename.

Tags matching `control-center-v*` build a named prerelease installer. In addition, every successful push to `main` publishes an installable `control-center-ci-<run>` prerelease with a SHA-256 checksum. Publication depends on frontend, Tauri, C++, injector, static-policy, gallery, hosted service, and installer gates. The release contains only the successful current installer; baseline and deliberate-failure fixtures remain CI-only. Pull requests retain read-only repository permissions and cannot publish releases.
